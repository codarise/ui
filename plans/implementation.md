# codarise-ui — Agent Implementation Guide

This is a **shadcn/ui registry** repo (`codarise/ui`). It distributes React components directly from GitHub source — no published bundle needed. Consumers run `npx shadcn@latest add codarise/ui/<name>` and the CLI pulls files from this repo.

The repo is also a **Vite showcase playground** that renders every registered item with live previews, search, and copy-import/install buttons.

### What's generated (don't edit)
- `src/generated/manifest.ts` — regenerated by `npm run gen:manifest` (auto-runs on `predev`/`prebuild`)
- `src/components/ui/` — shadcn CLI intermediate output (gitignored)
- `public/r/` — shadcn build output (gitignored)

### What's source (do edit)
- Everything under `registry/` — the canonical component source
- `registry.json` and `registry/<group>/registry.json` — the registry entries
- `src/components/showcase/` — the showcase app

## How to add a new item

### 1. Write the source file

Place it in the right directory based on type:

| Type | Directory | `type` field | Installs to (consumer) |
|------|-----------|-------------|----------------------|
| UI primitive | `registry/ui/` | `registry:ui` | `@/components/ui/<name>` |
| Custom primitive | `registry/custom/` | `registry:ui` | `@/components/ui/<name>` |
| Lib | `registry/lib/` | `registry:lib` | `@/lib/<name>` |
| Block | `registry/blocks/` | `registry:block` | `@/components/ui/<name>` |
| Theme/font | `registry/theme/` | `registry:theme` | CSS only |

### 2. Register it in the group's `registry.json`

Add an entry to the appropriate `registry/<group>/registry.json`:

```jsonc
{
  "name": "my-component",                    // kebab-case, matches filename
  "type": "registry:ui",                     // or registry:block / registry:lib
  "title": "My Component",                   // Title Case
  "description": "What it does, briefly.",
  "dependencies": ["lucide-react"],          // npm packages the consumer needs
  "registryDependencies": [                  // same-repo deps (full GitHub address)
    "codarise/ui/button",
    "codarise/ui/model-colors"
  ],
  "files": [
    { "path": "my-component.tsx", "type": "registry:ui" }
  ]
}
```

**`registryDependencies` rules:**
- Same-repo items: use the full GitHub address `codarise/ui/<name>`
- Official shadcn items: bare name (e.g. `"button"`)
- The CLI resolves and pulls these automatically

**`dependencies` vs `registryDependencies`:**
- `dependencies` = npm packages (e.g. `"lucide-react"`, `"radix-ui"`)
- `registryDependencies` = other registry items from this repo or shadcn

### 3. Regenerate the manifest

```bash
npm run gen:manifest
```

This reads all registry JSONs, extracts exports from each source file, and emits `src/generated/manifest.ts`. The item now appears in the showcase automatically. This also runs automatically before `dev` and `build`.

### 4. (Optional) Add a showcase preview

Open `src/components/showcase/previews.tsx`:

1. Add an import at the top (relative path from `src/components/showcase/`):
   ```ts
   import { MyComponent } from "../../../registry/ui/my-component"
   ```

2. Add a preview entry in the `previewComponents` map:
   ```tsx
   "my-component": () => (
     <MyComponent />
   ),
   ```

3. If the item has no meaningful visual (pure utility, type-only, theme), add it to `noPreviewItems` instead:
   ```ts
   export const noPreviewItems = new Set([
     "utils",
     "model-colors",
     // ...
   ])
   ```

Items without a preview render a tasteful placeholder card.

### 5. Verify

```bash
npm run typecheck                          # tsc --noEmit (must pass — noUnusedLocals + noUnusedParameters)
npm run build                              # gen:manifest → tsc -b → vite build
npm run lint                              # eslint (ignores registry/ and src/components/ui/)
npx shadcn@latest registry validate ./registry.json   # CI gate
```

All four must pass. CI (`.github/workflows/registry-validate.yml`) runs typecheck + registry validate + shadcn build on every PR.

## Code conventions

### fe-agentic is the source of truth for styling
When this repo's `new-york` defaults differ from the fe-agentic app (`/Users/flx/Documents/GitHub/fe-agentic/src/components/ui/`), **fe-agentic wins**. Examples:
- `Input` is `h-10` (not `h-9`)
- `Select` trigger is `h-10`, content is `h-9`
- `Tabs` uses simpler classes (no cva/variants)
- `Switch` has no `size` prop
- `Dialog` overlay: `bg-neutral-950/50 backdrop-blur-[1px]`, content: `sm:max-w-xl`, close button: `cursor-pointer`

Always diff against fe-agentic when syncing a component.

### Import rules

**Within `registry/` files (cross-component imports):**
- Use **relative** imports — never `@/components/ui/...`
- From a block: `../ui/button`, `../custom/copy-button`, `../lib/utils`
- From a UI primitive: `./button`, `./dialog`
- The `@/lib/utils` import is acceptable inside `registry/` files (the manifest generator and shadcn CLI both handle it — it maps to the consumer's `utils.ts`)

```tsx
// CORRECT — in registry/blocks/model-card.tsx
import { Card, CardContent } from '../ui/card'
import { getModelColorByName } from '../lib/model-colors'
import { cn } from '../lib/utils'

// WRONG — will break for consumers
import { Card } from '@/components/ui/card'
```

**Within `src/` (showcase app):**
- Use `@/` path alias (maps to `./src/`) or relative paths
- Registry imports use relative `../../../registry/...`

### Component style
- **Function components**, not `React.forwardRef` (new-york convention)
- `data-slot` attributes on all elements (e.g. `data-slot="button"`)
- `data-variant` / `data-state` / `data-size` for variant-aware styling
- Unified `radix-ui` import (not `@radix-ui/react-*` packages)
- `"use client"` directive at the top of client components (not strictly needed for Vite showcase but required for Next.js consumers)

```tsx
"use client"

import * as React from "react"
import { Dialog as DialogPrimitive } from "radix-ui"
import { cn } from "@/lib/utils"

function MyComponent({ className, ...props }: React.ComponentProps<"div">) {
  return (
    <div
      data-slot="my-component"
      className={cn("base-classes", className)}
      {...props}
    />
  )
}

export { MyComponent }
```

### TypeScript strictness
- `noUnusedLocals: true` — unused imports/variables **fail typecheck**
- `noUnusedParameters: true` — unused function params fail (prefix with `_` if intentional)
- `verbatimModuleSyntax: true` — use `import type { ... }` for type-only imports
- `strict: true`

## Common pitfalls

### 1. `@/` alias in registry files
Using `@/components/ui/button` inside `registry/` files will break for consumers — the CLI copies the file as-is, and the consumer may not have that exact path. Use relative imports (`./button`, `../ui/button`).

The one exception is `@/lib/utils` — this is a shadcn convention and the CLI/consumers expect it to resolve to their `utils.ts`.

### 2. Forgetting to regenerate the manifest
After adding an item to any `registry.json`, run `npm run gen:manifest` (or just `npm run dev` / `npm run build` which trigger it automatically). The showcase reads from the generated manifest, not the registry JSONs directly.

### 3. Unused imports after edits
When you remove a feature (e.g. the `showCloseButton` prop from `DialogFooter`), check if the imports it required (e.g. `Button`) are now unused. `noUnusedLocals` will fail the build.

### 4. `registryDependencies` format
- Wrong: `"button"` (bare name for same-repo items — this means official shadcn)
- Right: `"codarise/ui/button"` (full address for same-repo items)
- Right: `"button"` only if you mean the official shadcn button

### 5. File `type` field must match item `type`
For `registry:ui` items, each file entry should have `"type": "registry:ui"`. For blocks, `"type": "registry:block"`. The manifest generator uses the item-level `type` for install path derivation, but file-level `type` must be consistent for `registry validate`.

### 6. Blocks should be self-contained
Blocks must use relative imports (`../ui/`, `../custom/`, `../lib/`) for portability. App-specific coupling (router `Link`, contexts, API clients) must be decoupled — either use plain `<a>` tags (like `sidebar` block) or accept the dependency as a prop (like `page-layout` block's `backButton` prop).

## Syncing a component from fe-agentic

When porting or syncing a component from the fe-agentic app:

```bash
# 1. Diff the two versions
diff /Users/flx/Documents/GitHub/fe-agentic/src/components/ui/<name>.tsx \
     /Users/flx/Documents/GitHub/codarise-ui/registry/ui/<name>.tsx

# 2. fe-agentic styling wins — apply its classes
# 3. Keep new-york structural conventions (function components, data-slot, radix-ui import)
# 4. Fix any @/components/ui/ imports to relative ./
# 5. Verify: typecheck + build + lint + registry validate
```
